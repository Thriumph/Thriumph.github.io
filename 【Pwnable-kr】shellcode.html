【Pwnable.kr】shellcode

  • pwn
  • pwnable.kr

题目源码

0x0A

源码分析

用了两个setresuid函数，也就是为shellshock可执行文件设置组用户对flag文件的读权限

之前不知道getegid函数,百度一波，大概是跟权限有关的

getegid()用来取得进程执行有效组识别码。有效的组识别码用来决定进程执行时组的权限。返回有效的组识别码。

查了一下shellshock,涉及一个关于bash漏洞CVE-2014-6271

https://www.freebuf.com/news/48331.html

Shellshock，又称Bashdoor，是在Unix中广泛使用的Bash shell中的一个安全漏洞，首次于2014年9月24日公开。许多互联网守护进程，如网页服务器，使用bash来处理某些命令，从而允许攻击者在易受攻击的Bash版本上执行任意代码。这可使攻击者在未授权的情况下访问计算机系统

漏洞原因

Bash使用的环境变量是通过函数名称来调用的,导致漏洞出问题是以“() {”开头定义的环境变量在命令ENV中解析成函数后,Bash执行并未退出,而是继续解析并执行shell命令。而其核心的原因在于在输入的过滤中没有严格限制边界,也没有做出合法化的参数判断。

本地检测

env x='() { :;}; echo vulnerable' bash -c "echo this is a test"
这个漏洞存在

利用

echo vulnerable后可以执行任意代码

该漏洞影响的bash使用的环境变量是通过函数名称来调用的，以“(){”开头通过环境变量来定义的。而在处理这样的“函数环境变量”的时候，并没有以函数结尾“}”为结束，而是一直执行其后的shell命令，env x= ‘’表示向环境变量中注入x = ….的变量，bash在处理这样以”(){“开头的环境变量的值时并没有以函数结尾”}”为结束，而是一直执行其后的shell命令，最后就会执行”echo vulnerable”这个语句

env x='() { :;}; ' bash -c "cat flag"

发现没有权限，查看权限

启动时，用户是shellshock权限，也就是other权限，这样flag是不可读的，但是shellshock这个可执行文件与root用户是在同一个组中，对group是具有特殊权限读(w)的，所以egid是属于shellshock_pwn的，这个权限很高，对flag文件是可读的-r-xr-sr-x表示SGID和同组用户权限中可执行位被设置

echo vulnerable’ 换成想执行的命令 bash -c “cat ./flag”

./ bash -c “echo this is a test” 换成./shellshock

env var='() { :;}; cat flag' ./shellshock

only if I knew CVE-2014-6271 ten years ago..!!
